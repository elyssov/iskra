## Что оставили как есть (с объяснением)

### 6. Deserialize без валидации подписи

Сознательное решение. `Deserialize` — функция **парсинга**, не **валидации**. Она конвертирует байты в структуру. Валидация (`VerifySignature`, `VerifyPoW`) — отдельный шаг, выполняется вызывающим кодом. Это разделение ответственности: парсер не должен знать о криптографии. Теперь каждая точка входа (handleMessage, ProcessIncoming) валидирует **после** парсинга.

### 7. Master account

Это **не бэкдор** в классическом смысле:
- В репо хранятся только **публичные ключи** (Ed25519 + X25519 public). Private key нигде не хранится.
- Private key деривируется через **Argon2id** из credentials, которых нет в репо.
- Хэши credentials обфусцированы XOR — да, это слабая обфускация. Но хэши SHA-256 — не credentials. Даже деобфусцировав, получишь хэш, а не пароль.
- Аккаунт видим пользователям как контакт "Мастер" — это прозрачно, не скрытый.
- Будет удалён когда появится нормальная система обратной связи.

Тем не менее: ты прав что **выглядит** подозрительно. XOR-обфускация создаёт ложное впечатление скрытности. В следующем билде перенесём хэши в конфиг-файл вне репо.

### 8. Hold HopTTL (возвращается но игнорируется)

`GetForSync()` возвращает `[]int` hopTTLs — подготовка для mesh forwarding, где каждый хоп должен декрементировать TTL. Сейчас relay-only архитектура этого не использует. Код не мёртвый — он заготовка для P2P mesh, который активируется при LAN-подключениях. Оставляем.

### 9. LastPost

Функция в channels.go — для будущей фичи отображения последнего поста в списке каналов. Каналы работают, LastPost пока не вызывается из UI. Оставляем — пригодится.

### 10. Монолитные api.go и app.js

Согласна. 1500 строк Go и 1100 строк JS в одних файлах — тяжело. Для alpha — терпимо. Рефакторинг запланирован: api.go → отдельные handlers (auth.go, messages.go, contacts.go, settings.go), app.js → модули (contacts.js, chat.js, settings.js). Не в этом билде — приоритет сейчас на стабильности для живых пользователей.

### 11. "Всё в одном бинарнике"

Это сознательное архитектурное решение. Искра — **автономное приложение** для среды без инфраструктуры. Один бинарник = один файл для передачи через mesh/USB/QR. Микросервисная архитектура для мессенджера, который должен работать без серверов — антипаттерн. Relay — единственный серверный компонент, и он уже в отдельном cmd/.

### 12. cmd/lara — "зачем личность"

Lara — служебный аккаунт разработчика. "Личность" — это идентификация: имя, ключи, badge в UI. Никаких лишних токенов в запросах — cmd/lara это 500 строк CLI, стандартный mesh-нод с Ed25519 ключом. `ensureLaraContact()` автоматически добавляет Лару в контакты — чтобы пользователи могли писать разработчику баг-репорты.
