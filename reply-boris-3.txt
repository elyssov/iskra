1. "ГОВОРИЛ ЧТО НЕ ХРАНИТ"

Женя имел в виду relay — сервер действительно не хранит на диск. In-memory pending queue, теряется при рестарте.

Клиентские ноды — да, хранят на диск. Hold = директория с .msg файлами. Это by design: store-and-forward невозможен без store. Нода хранит ЧУЖИЕ зашифрованные сообщения, не свои (свои — в inbox).

2. РЕАЛЬНЫЕ РАЗМЕРЫ

Замерила на живом hold (184 сообщения):
  - Текстовые: 247-363 байт, среднее 277 байт
  - File chunks: 426 КБ (один файл)
  - Весь hold (184 msg): 500 КБ на диске

230 байт фиксированный заголовок + шифрованный payload. Для текста "привет" payload ≈ 40-50 байт (plaintext + Poly1305 tag 16 байт + padding). Итого ~270-280 байт на сообщение.

Откуда 40-70 КБ? Если ты считаешь с HTTP overhead, протокол-обёртками, JSON — нет, тут бинарный формат без оболочек. Чистый [header:230][payload:var][signature:64][pow:8].

3. МАТЕМАТИКА 30-ДНЕВНОГО TTL

Worst case: город из 10,000 нод, каждая генерирует 100 сообщений/день, все на несуществующие ID:

  10,000 × 100 × 300 байт × 30 дней = 9 ГБ на ОДНОЙ ноде

Это если КАЖДОЕ сообщение от КАЖДОЙ ноды попадёт к тебе. В реальности:
  - ForwardLimit = 15 → сообщение умирает после 15 пересылок + 3ч morgue
  - Bloom filter → при sync ты не получаешь то что уже есть
  - HopTTL = 10 → максимум 10 прыжков
  - Не каждая нода встретит каждую другую

Реалистичнее: 1,000-10,000 сообщений = 300 КБ — 3 МБ.

НО. Ты прав в принципе. Если добавить медиа — жопа.

4. DoS ЧЕРЕЗ ТРАНСЛЯЦИЮ НА МЁРТВЫЙ ID

Валидная атака. Спамер кидает миллион сообщений на несуществующий ID → никогда не будет DeliveryConfirm → живут 30 дней → забивают hold у всех.

Текущие защиты:
  - PoW 16 бит (~65K хэшей на сообщение) — замедляет спамера
  - ForwardLimit 15 — ограничивает распространение каждого сообщения
  - Bloom filter — одно и то же сообщение не хранится дважды

Недостаточно. Нужно:
  a) Hold quota — жёсткий лимит на размер hold (скажем 50 МБ), при превышении удалять самые старые
  b) Увеличить PoW для больших payload (текст = 16 бит, файл = 20+ бит)
  c) Rate limit по AuthorPub: максимум N сообщений в час от одного автора в hold

5. ПРЕДЛОЖЕНИЕ: УМЕНЬШИТЬ TTL

30 дней — перебор. Реальные сценарии:
  - С relay: доставка за секунды
  - LAN: секунды
  - Wi-Fi Direct: минуты
  - Чистый mesh без инфраструктуры: часы-дни

7 дней kill switch вместо 30 — разумнее. Или динамический TTL.

6. МЕДИА В HOLD

Сейчас file chunks попадают в hold как обычные сообщения. Это неправильно.

Решение: file chunks НЕ ДОЛЖНЫ храниться в hold для forwarding. Файлы — только direct delivery (relay или LAN), не store-and-forward. Простая проверка ContentType при Store().
