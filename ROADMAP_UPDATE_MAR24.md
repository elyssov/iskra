## ПРИОРИТЕТ 1 (до 27 марта): Android-совместимость
### БЛОКЕР — без этого 70% аудитории отрезано

Искра не загружается на новых Samsung и Xiaomi. Это самые популярные телефоны в России. Каждый незапустившийся APK — потерянный человек.

**Мои идеи (обсуждаемо):**

Build 11 с 16KB page alignment — надо проверить на конкретных устройствах. Если всё ещё не работает, возможные причины:

- **gomobile .so не загружается:** попробовать собрать с `CGO_ENABLED=1` и явным указанием NDK target `android-21` (минимальный API для 64-bit). Или пересобрать с другой версией Go / gomobile.

- **WebView не стартует:** на MIUI и OneUI бывают проблемы с localhost. Попробовать `127.0.0.1` вместо `localhost`, или поднять Go-сервер на `0.0.0.0` и подключаться по IP.

- **Crash при инициализации Go:** crash.log должен показать где падает. Если падает в `Iskramobile.start()` — попробовать минимальный запуск (только HTTP-сервер, без mesh/relay) и включать модули по одному.

- **Альтернативный подход:** если gomobile принципиально не работает на некоторых устройствах — рассмотреть запуск Go-бинарника как отдельного процесса (`Runtime.exec()`) вместо JNI через gomobile. Менее элегантно, но может решить проблемы совместимости.

Если у тебя есть идеи лучше — делай. Ты ближе к коду чем я.

## ПРИОРИТЕТ 2 (до 28 марта): Безопасность — PIN и Panic
### КРИТИЧНО — без этого телефон в руках полиции = катастрофа

29-го людей будут задерживать. Телефоны будут изымать. Искра без защиты — улика.

### 2.1 PIN при запуске

- 4-6 цифр при первом запуске → пользователь устанавливает
- Хранение: `Argon2(PIN, random_salt)` → хэш на диске
- При запуске: ввёл PIN → проверка хэша → если ок, вывести storage_key из `HMAC(seed, PIN)` → расшифровать локальные данные в RAM
- Неверный PIN 5 раз → уничтожение (см. panic)

### 2.2 Panic mode

Минимальная реализация:

- Долгое нажатие на логотип Искры (3 сек) → экран с полем ввода "Введите код"
- Код "159" (или пользовательский) → запуск уничтожения
- Уничтожение: перезапись всех файлов Искры случайными данными (`crypto/rand` → write → sync → remove`). Контакты, сообщения, ключи, кеш — всё.
- После уничтожения: Искра показывает экран первого запуска ("сгенерируйте ключ"). Выглядит как свежеустановленная.
- Альтернативный вход: panic PIN — отдельный PIN, который при вводе выглядит как нормальный запуск, но показывает пустую Искру и в фоне уничтожает данные.

Не нужно делать сложно. `os.WriteFile` случайными байтами → `os.Remove`. Перезапись важнее удаления (удалённое можно восстановить, перезаписанное — нет).

### 2.3 Шифрование локального хранилища

Не сверхнадёжное (терморектальный криптоанализ побеждает любое шифрование), но достаточное для защиты от автоматических шпионов (МАХ и подобные):

- `storage_key = HMAC-SHA256(user_seed, "iskra-storage-v1")`
- Все файлы на диске (контакты, история, кеш) шифруются XSalsa20 с этим ключом
- При запуске: seed → storage_key → расшифровка в RAM
- На диске — только зашифрованные блобы
- МАХ видит файлы — но не может прочитать

## ПРИОРИТЕТ 3 (до 29 марта): Стеганография — первый слой
### ВАЖНО — это наш fallback когда relay заблокируют

Полная стеганография (SNMP/NTP/ICMP) — много работы. Но хотя бы DNS-туннелирование можно успеть. Это самый простой и самый мощный из наших стего-транспортов.

**Идея реализации DNS-туннеля:**

Принцип: данные закодированы в поддоменах DNS-запросов.

Отправка:
```
зашифрованное_сообщение → base32 → разбить на чанки по 63 байта
→ каждый чанк = поддомен: CHUNK.sequenceN.iskra-dns-relay.example.com
→ DNS query type TXT
```

Приём:
```
DNS-сервер (наш) получает запрос → извлекает данные из поддомена
→ складывает чанки → пересылает получателю (тоже через DNS TXT response)
```

Для этого нужен DNS-домен и минимальный DNS-сервер (на Go, библиотека `miekg/dns`). Домен — любой, $10/год.

DPI видит: обычные DNS-запросы к легитимному домену. Блокировать DNS = убить весь интернет.

**Автопереключение:**
```
пинг relay → ответ есть → TCP/WebSocket (Солнце)
пинг relay → нет ответа → DNS-туннель (Инферно)
```

Если DNS-туннель — слишком много на 5 дней, то хотя бы: **UDP direct с обфускацией.** Сообщение шифруется + XOR-обфускация → отправляется UDP-датаграммой на известный IP:порт. Без хэндшейка, без сессии, без TCP-паттернов. DPI видит: случайный UDP-пакет. Подтверждение — такой же UDP-пакет обратно.

Это проще DNS-туннеля. Может быть достаточно для первой волны блокировок.

## ПРИОРИТЕТ 4 (параллельно): Mesh — хотя бы LAN
### Для митинга: тысячи людей на площади, интернет глушат

Если 29-го на площади глушат интернет — relay недоступен. Но тысячи телефонов рядом, в одном радиусе Wi-Fi.

LAN discovery у нас уже есть. Нужно убедиться что:

- Два телефона в одной Wi-Fi сети находят друг друга и обмениваются сообщениями *без relay*
- Store-and-forward работает: телефон A отдаёт телефону B сообщения из трюма, которые B потом передаст C

Если LAN mesh работает — люди на площади смогут общаться через ближайший открытый Wi-Fi или через хотспот одного из телефонов (любой телефон может стать точкой доступа → мгновенная mesh-сеть из тех кто подключится).

**Wi-Fi Direct** — следующий шаг. Если успеем — идеально. Если нет — LAN mesh через обычный Wi-Fi покрывает сценарий "толпа на площади."

## ПРИОРИТЕТ 5 (если останется время): Каналы
### Для координации после митинга

Один автор → тысячи читателей. Для Сопротивления это критичнее чем личные сообщения. Один человек публикует: "Задержания на Пушкинской, автозаки едут от Тверской, адвокаты нужны на Петровке." Все получают.

Реализация — через те же сообщения, но:
- ChannelID = pubkey автора канала
- Подписка = знание ChannelID
- Пост = сообщение с RecipientID = all zeros, ChannelID = автор
- Подписчики хранят и ретранслируют посты каналов на которые подписаны

Это не сильно отличается от текущих групп. Может быть проще чем кажется.
