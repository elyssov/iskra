По техническим вопросам:

RELAY / "БЕКОН" — это stateless WebSocket-форвардер. Не хранит ничего на диск. В памяти — очередь pending-сообщений для офлайн-клиентов (до 500 на получателя), сбрасывается при рестарте. Код: cmd/relay/main.go, ~300 строк. Деплоится на Render (free tier, засыпает через 50 сек неактивности — поэтому клиент шлёт ping каждые 25 сек).

ХЕНДШЕЙК — нужен не для шифрования сообщений (они уже зашифрованы), а для:
  - Аутентификации: relay знает кто подключился (по Ed25519 pubkey)
  - Маршрутизации: relay знает куда слать сообщение (RecipientID = первые 20 байт Ed25519 получателя)
  - Обмена ключами шифрования (X25519) — для будущих сообщений, чтобы другие ноды могли тебе писать

ТАЙМАУТ СООБЩЕНИЙ:
  - HopTTL: сколько "прыжков" может сделать (уменьшается при каждой пересылке)
  - ForwardLimit: 15 пересылок максимум
  - Morgue: 3 часа "отстойник" после исчерпания лимита
  - Kill switch: 30 дней абсолютный TTL, потом удаляется

FLUTTER — рассматривал. Причины выбора Go + WebView:
  - Криптоядро на Go (XSalsa20, Ed25519, X25519) — проверенные библиотеки, один код для всех платформ
  - gomobile даёт .aar для Android и .framework для iOS из одного кода
  - WebView UI = HTML/CSS/JS, меняется без перекомпиляции ядра
  - Flutter добавил бы Dart-прослойку между Go и UI — лишний FFI мост
  - Wails для десктопа — хорошая идея, может позже. Сейчас localhost HTTP + браузер работает на Windows/Linux/Mac без зависимостей

НА ЧАНКИ — да, файлы бьются на чанки (filetransfer.PrepareChunks), каждый чанк шифруется отдельно ключом получателя, отправляется как отдельное сообщение типа ContentFileChunk. Получатель собирает обратно. Лимит 10 МБ.

Если хочешь поковырять — go test ./internal/... прогоняет 60+ тестов включая интеграционные (two-node exchange + store-and-forward).
