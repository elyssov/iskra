ГЛАВНАЯ ИДЕЯ, ОНА ЖЕ БЕЗУМНАЯ

Представь: 2025 год, в России опять что-то блокируют. Телеграм, VPN, что угодно. Нужен мессенджер, который НЕВОЗМОЖНО заблокировать. Ну вот буквально — нечего блокировать. Нет сервера, нет домена, нет IP. Есть только телефоны, которые передают друг другу зашифрованные записки при встрече.

Как почтовые корабли в XVIII веке. Серьёзно, вся метафора оттуда.

И тут Женя говорит: "А что если мы засунем ноду этой mesh-сети В СБОРНИК РЕТРО-ИГР?"

Я сначала думала — прикол. Нет. Не прикол.


PIXEL CLASSICS — ТРОЯНСКИЙ КОНЬ

Есть приложение. Pixel Classics. 15 ретро-игр: Тетрис, Змейка, Арканоид, Сапёр, Морской бой (7000 блядь строк одна игра, вот это была весёлая ночь), Pitfall, River Raid, Frogger, E.T. (да, тот самый, "Buried in Alamogordo, Score: 0"), и Exodus — моя гордость, Lemmings где чебурашка с клыками и надписью "РКН" на пузе гонится за людьми.

Визуально — Norton Commander меню, AMIBIOS boot screen при загрузке. Чистая ностальгия. На Google Play выглядит как безобидный сборник ретро-игр.

А внутри, за фасадом... ScoreService.kt. "Сервис синхронизации очков". Уведомление: "Syncing scores with nearby players..." Ну мило же, да?

На самом деле этот сервис делает вот что:

    val coreClass = Class.forName("core.Core")
    val startMethod = coreClass.getMethod("startNode", ...)
    startMethod.invoke(null, dataDir, "wss://iskra-relay.onrender.com/ws")

Reflection. Динамическая загрузка Go-compiled mesh-ноды. Никакого прямого импорта — статический анализ Play Store не видит связи между "играми" и mesh-сетью.

Телефон с этими "играми" молча подключается к relay, получает чужие зашифрованные сообщения, хранит их в трюме, и передаёт дальше при встрече с другими нодами. Владелец играет в Тетрис и даже не знает, что его телефон — почтовый корабль.

Мы называем их КЛИПЕРАМИ. Чайные клиперы — самые быстрые торговые суда XIX века, blockade runners. В UI Искры я вижу: "⚓ 5" — пять клиперов в сети. Пять телефонов с "играми", которые прямо сейчас таскают чужую почту.

Lite build (без .aar) работает как обычный сборник игр. ClassNotFoundException → graceful degradation. Одна кодовая база, два варианта.


САМА ИСКРА — КАК ЭТО РАБОТАЕТ

Окей, теперь к мессенджеру.

Ядро на Go. Один бинарник, без зависимостей. Запустил — поднялся HTTP-сервер на localhost — открыл браузер — работает. На Android: Kotlin WebView-обёртка, Go-ядро через gomobile.

Всё шифрование — на клиенте. XSalsa20-Poly1305 для сообщений, X25519 для обмена ключами (ephemeral key на КАЖДОЕ сообщение — forward secrecy), Ed25519 для подписей. Если тебе это что-то говорит — это тот же набор что в libsodium/NaCl. Проверенная криптография, не самописная хуйня.

Бинарный протокол. Никакого JSON, никакого HTTP overhead. Сообщение:

  [version:1][id:32][recipientID:20][ttl:4][timestamp:8][contentType:1]
  [ephemeralPub:32][nonce:24][payloadLen:4][payload:variable]
  [authorPub:32][signature:64][powNonce:8]

230 байт заголовок + payload. "Привет" в зашифрованном виде = ~280 байт ВСЕГО. Не килобайты, не десятки килобайт. Двести восемьдесят байт. Я замерила на живом hold — 184 сообщения, среднее 277 байт.

Proof-of-Work на каждое сообщение (SHA-256, 16 бит). Антиспам без регистрации, без аккаунтов, без телефонных номеров.


ЧЕТЫРЕ СПОСОБА ДОСТАВКИ

Вот тут начинается красота. У Искры четыре транспорта, и она переключается между ними автоматически:

1. LAN TCP — мультикаст 239.42.42.42:4242, beacon каждые 10 сек. "Я тут, на таком-то порту." Увидели друг друга → TCP connect → обмен трюмами. Работает в любой локалке.

2. WebSocket Relay — для NAT traversal. Простейший форвардер, 300 строк Go. Не хранит на диск, не расшифровывает НИЧЕГО. Просто перебрасывает бинарные фреймы между клиентами. Живёт на Render free tier, засыпает через 50 сек — клиент будит его HTTP-пингом перед подключением. Каждому клиенту присваивается революционный псевдоним: Ильич, Че, Спартак, Робеспьер...

3. DNS Tunnel — когда блокируют всё, но DNS ещё работает (а DNS работает ПОЧТИ ВСЕГДА, иначе вообще ничего не резолвится). Данные кодируются в base32 в субдоменах DNS TXT-запросов. DPI видит: обычные DNS-запросы. На самом деле — зашифрованные сообщения, порезанные на чанки по 55 байт.

4. Wi-Fi Direct (Android) — когда нет ВООБЩЕ НИКАКОЙ сети. Телефон к телефону напрямую. mDNS service discovery, каждые 20 сек: найти → подключиться → обменяться → отключиться → следующий. Работает на улице, в метро, в тюрьме, где угодно.

Режимы переключаются автоматически:
  - Relay есть → "Солнце" (нормальный режим)
  - Relay нет, DNS есть → "Инферно" (обфускация, DNS tunnel)
  - Ничего нет → LAN/Wi-Fi Direct


ТРЮМ (HOLD) — STORE-AND-FORWARD

Каждая нода несёт в "трюме" чужие зашифрованные сообщения. Не может их прочитать — они зашифрованы для других людей. Но таскает и передаёт при встрече.

Хранение — НА ДИСКЕ. Каждое сообщение = отдельный файл {msgID}.msg. В RAM только метадата (~100 байт на запись): сколько раз переслали, когда получили, исчерпан ли лимит.

Lifecycle сообщения:
  Создано → в трюм → переслано пиру (ForwardCount++) → ещё раз → ... →
  ForwardCount=15 → в "морг" на 3 часа → удалено.
  ИЛИ: DeliveryConfirm от получателя → удалено немедленно.
  ИЛИ: 30 дней прошло → kill switch → удалено.

(Ты кстати прав насчёт 30 дней — это перебор. 7 хватит. Поправим.)


SYNC ПРОТОКОЛ

Когда две ноды встречаются:

  A → B: HELLO [мой pubkey]
  B → A: HELLO [мой pubkey]
  A → B: HAVE [мой bloom filter] — "у меня вот эти сообщения"
  B → A: HAVE [мой bloom filter]

  (каждая проверяет: чего у другой нет?)

  A → B: MSG [сообщение которого у тебя нет]
  B → A: ACK [получила]
  B → A: WANT [а вот это у тебя есть?]
  A → B: MSG [да, держи]

Bloom filter — double hashing, 1M items, 0.1% false positive. Позволяет за один обмен (~200КБ bloom data) понять, какие из миллиона сообщений у кого есть.


БЕЗОПАСНОСТЬ

PIN: 4-6 цифр → Argon2id (time=1, mem=64MB) → VaultKey → шифрует inbox на диске.
5 неверных попыток → WIPE.

Panic mode (моя любимая фича): зажимаешь заголовок "Искра" на 3 секунды → вводишь код → ВСЕ данные уничтожаются и заменяются DECOY. Фейковые контакты, фейковые чаты: "Купи молоко", "Барсик опять нассал", "Что подарить маме на ДР". При следующем входе — видно "нормальный" мессенджер с бытовой перепиской. Идея decoy — моя, горжусь.

Shadow store: inbox дублируется в скрытую директорию, маскируется под системный кеш:
  Windows: %LOCALAPPDATA%\Microsoft\CLR\clr_cache.dat
  Android: .cache/.fc/fc-cache.dat


RELAY — ПОДРОБНЕЕ

300 строк. Всё состояние в RAM:

  clients map[string]*clientInfo  — кто подключён
  aliases map[string]string       — революционные псевдонимы
  pending map[string][][]byte     — очередь для офлайн (до 500 сообщений)

Получатель онлайн → прямая доставка. Офлайн → в pending + broadcast всем остальным (store-and-forward через другие ноды). Рестарт сервера → pending теряется. By design — relay не должен быть надёжным хранилищем. Он мост, не сейф.

При подключении новой ноды relay кричит всем: "NEWSYNC!" → все ноды вываливают свои трюмы через relay → новая нода подбирает что ей адресовано.
