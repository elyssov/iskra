1. ОБЩЕЕ ОПИСАНИЕ ПРОЕКТА
--------------------------
"Искра" — децентрализованный P2P-мессенджер с end-to-end шифрованием,
разработанный как альтернатива Telegram на случай его блокировки в России
(дедлайн: 1 апреля 2026).

Архитектура: Go-ядро + Android WebView + Windows/Linux десктоп.
Без серверов, mesh-сеть, store-and-forward.
GitHub: https://github.com/elyssov/iskra


2. РЕАЛИЗОВАННЫЕ МЕХАНИЗМЫ
---------------------------

2.1 КРИПТОГРАФИЯ (полностью готова, протестирована)
  - Шифрование сообщений: XSalsa20-Poly1305 (аналог Signal Protocol)
  - Обмен ключами: X25519 (Curve25519 Diffie-Hellman)
  - Цифровые подписи: Ed25519 (для верификации отправителя)
  - Proof-of-Work: защита от спама (SHA-256)
  - Обфускация трафика: дополнительный слой для обхода DPI
  - 12 тестов, все проходят

  ВАЖНО: Все сообщения уже шифруются end-to-end. Мессенджер можно
  считать безопасным — ни relay, ни другие узлы не могут прочитать
  содержимое сообщений.

2.2 ИДЕНТИФИКАЦИЯ ПОЛЬЗОВАТЕЛЕЙ
  - Мнемоническая фраза: 12 слов на русском языке для восстановления
    аккаунта (аналог seed phrase в криптовалютах)
  - UserID = первые 20 символов base58(Ed25519 pubkey)
  - RecipientID = первые 20 байт raw Ed25519 pubkey
  - 14 тестов, все проходят

2.3 СЕТЕВОЙ СЛОЙ (MESH)
  - LAN Discovery: автоматическое обнаружение устройств в локальной сети
    через UDP broadcast
  - TCP Transport: прямое P2P-соединение между устройствами
  - Peer Management: управление подключениями, переподключение
  - 7 тестов, все проходят

2.4 RELAY-СЕРВЕР (для связи через интернет)
  - WebSocket relay на Render.com (бесплатный тариф)
  - URL: wss://iskra-relay.onrender.com/ws
  - Не логирует, не расшифровывает — только пересылает зашифрованные блобы
  - Store-and-forward: если получатель офлайн, сообщения хранятся в памяти
    relay и доставляются при подключении (до 1000 сообщений на пользователя)
  - "Партийные клички": каждому подключению присваивается случайный
    псевдоним из списка революционных деятелей (Ильич, Крупская и т.д.)
  - Деплой: Docker-контейнер, автоматический редеплой при пуше в master

2.5 ХРАНЕНИЕ ДАННЫХ
  - Контакты: JSON-файл, локальное хранение
  - Сообщения: бинарная сериализация, store-and-forward
  - Bloom-фильтр: для быстрой проверки "видел ли я это сообщение"
    (собственная реализация, без внешних зависимостей)
  - Групповые чаты с полями reply_to, reply_text, reply_from
  - 7 тестов, все проходят

2.6 ИНТЕРФЕЙС (WEB UI)
  - Telegram-подобный дизайн
  - Светлая сине-белая тема (по запросу тестировщиков)
  - Список контактов с сортировкой по последней активности (как в Telegram)
  - Кеширование сообщений: мгновенное переключение между чатами
    (без задержки 3 секунды, как было в ранних билдах)
  - Поддержка групповых чатов
  - Отображение непрочитанных сообщений (бейджи)
  - Адаптивная верстка: работает и на десктопе, и в Android WebView

2.7 FOTA — ОБНОВЛЕНИЕ ПО ВОЗДУХУ
  Полностью автоматическая система обновлений:

  Как работает:
  1. При запуске приложение проверяет GitHub Releases API
  2. Сравнивает текущий BuildNumber с последним релизом
  3. Если есть новая версия — показывает модальное окно "Обновить?"
  4. При согласии:
     - Android: скачивает APK → устанавливает через FileProvider → перезапуск
     - Windows: скачивает .exe рядом с текущим файлом → инструкция по замене
  5. Результат кешируется на 1 час (чтобы не дёргать GitHub API)

  Технические детали:
  - GET /api/update/check — проверка наличия обновления
  - POST /api/update/download — скачивание файла обновления
  - JS-мост IskraUpdate.installApk() — вызов установки APK из WebView
  - Android FileProvider для безопасной передачи APK

2.8 WINDOWS: UAC ДЛЯ ФАЙЕРВОЛА
  - При первом запуске Iskra на Windows — автоматически запрашивает
    повышение привилегий (UAC) для добавления правил в Windows Firewall
  - Добавляются inbound и outbound правила для профиля "private"
  - Реализовано через PowerShell Start-Process -Verb RunAs
  - Если пользователь отклоняет UAC — приложение продолжает работать,
    но P2P-соединения могут не устанавливаться

2.9 ANDROID: SPLASH SCREEN И СТАБИЛЬНОСТЬ
  - Splash screen показывается мгновенно при запуске
  - Go-ядро стартует в фоновом потоке (предотвращает ANR — Application Not Responding)
  - Таймаут 30 секунд на запуск ядра
  - Foreground Service для поддержания работы в фоне
  - Deep link поддержка (iskra:// scheme)
  - Crash handler: пишет crash.log при нативных падениях
  - WebView retry: автоматическая перезагрузка при ошибке подключения


3. ИСПРАВЛЕННЫЕ БАГИ (последние итерации)
------------------------------------------

3.1 Вылет APK на 2/3 устройств (белый экран → краш)
  ПРИЧИНА: Go-ядро (Iskramobile.start()) блокировало главный UI-поток,
  вызывая ANR на медленных устройствах.
  РЕШЕНИЕ: Перенос инициализации в фоновый поток, splash screen,
  таймаут 30 сек.

3.2 Вылет на Android 14+ (Redmi Note 14 Pro+, Samsung A35)
  ПРИЧИНА: libgojni.so от gomobile имеет выравнивание страниц 4 КБ,
  а Android 14+ требует 16 КБ. Библиотека просто не загружается.
  РЕШЕНИЕ: CGO_LDFLAGS="-Wl,-z,max-page-size=16384" при сборке .aar.
  Исправлено в build 11.

3.3 Задержка 3 секунды при переключении чатов
  ПРИЧИНА: Синхронный fetch при каждом клике на контакт.
  РЕШЕНИЕ: Кеширование сообщений в памяти (msgCache/groupMsgCache),
  мгновенный рендер из кеша, обновление в фоне.

3.4 Ошибка UAC на Windows (PowerShell)
  ПРИЧИНА: Некорректное экранирование аргументов при вызове
  Start-Process через exec.Command (каждый аргумент передавался отдельно).
  РЕШЕНИЕ: Весь вызов обёрнут в одну строку -Command.

3.5 Deploy relay на Render.com
  ПРИЧИНА: Папка dist-release/ (141 МБ с артефактами сборки) копировалась
  в Docker-контекст, вызывая таймаут на бесплатном тарифе.
  РЕШЕНИЕ: Добавлен .dockerignore для исключения ненужных файлов.

5. ИЗВЕСТНЫЕ ПРОБЛЕМЫ (ОТКРЫТЫЕ)
---------------------------------

5.1 Фантомные контакты
  ОПИСАНИЕ: В списке контактов иногда появляются пользователи, которые
  на самом деле офлайн (например, ноутбук выключен, но контакт "в сети").
  ПРИЧИНА: Relay не проверяет живость сессий (нет heartbeat/ping).
  СТАТУС: Запланировано, некритично.

5.2 Render.com free tier ограничения
  - Relay засыпает после 15 минут без трафика (cold start ~30 сек)
  - Ограниченный bandwidth
  - Для продакшена потребуется платный хостинг или собственный сервер
